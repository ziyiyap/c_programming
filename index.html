Day 01 Setting Up Your Environment Beginner

Install GCC (or Clang), configure a code editor, and compile your first C program. Understanding the compilation pipeline — source → preprocessor → compiler → assembler → linker → binary — is fundamental. Unlike Python's interpreter, C code must be explicitly compiled before running.

🐍

From PythonPython runs with python3 hello.py. C requires gcc -Wall -o hello hello.c first, then ./hello. The extra step gives you a faster, standalone binary.

#include <stdio.h>

int main() {
    printf("Hello, C!\n");
    return 0;
}
// Compile: gcc -Wall -o hello hello.c
// Run:     ./hello

  3. Modify it to print your name, the current year, and a favourite quote on separate lines.
  4. Intentionally break the code (remove a semicolon, a #include, or the return). Read the compiler error carefully. Fix it.
  5. Compile with -Wall -Wextra flags. Note what extra warnings appear.

Day 02 Data Types & Variables Beginner

C is statically typed — every variable must declare its type before use. Unlike Python, types have fixed sizes and ranges. Understanding int, float, double, char, and their unsigned variants is non-negotiable.

🐍

From PythonPython's x = 5 auto-detects the type. In C, you write int x = 5; — type first, always. And unlike Python integers, C integers can overflow silently!

int    age    = 25;          // -2^31 to 2^31-1
long   big    = 9876543210L;
float  pi     = 3.14f;        // ~7 decimal digits
double precise = 3.14159265; // ~15 decimal digits
char   letter = 'A';          // single character
unsigned int u = 4000000000U;// no negatives

printf("int: %d bytes\n", (int)sizeof(int));

Exercises

  1. Print the size (in bytes) of every fundamental type using sizeof(): char, short, int, long, long long, float, double.
  2. Find the max value of int by including <limits.h> and printing INT_MAX and INT_MIN.
  3. Demonstrate integer overflow: start with INT_MAX, add 1, and print the result. Explain what happened.
  4. Declare a char variable holding 'Z'. Print it as a character, as its ASCII integer value, and add 1 to it (what letter do you get?).
  5. Write a program that converts a temperature in Celsius (stored as double) to Fahrenheit and Kelvin.

Day 03 Input/Output with printf & scanf Beginner

printf and scanf are C's primary I/O functions. You must match format specifiers precisely to variable types — a mismatch is undefined behaviour. scanf reads from stdin; & (address-of) is required for most types.

int age; double height; char name[50];
printf("Enter name: ");
scanf("%49s", name);       // %49s limits input length
printf("Enter age and height: ");
scanf("%d %lf", &age, &height); // note & before age, height
printf("%-10s | %3d | %.2f\n", name, age, height);

Exercises

  1. Write a program that reads two integers and prints their sum, difference, product, and quotient (handle division by zero).
  2. Read a full name with spaces using fgets(name, sizeof(name), stdin) and print it back.
  3. Practice format specifiers: print an integer in decimal, octal (%o), and hex (%x).
  4. Print a neatly aligned table of squares and cubes for numbers 1–10 using width specifiers like %5d.
  5. Debug this broken code: int x; scanf("%d", x); — what's wrong? Fix it.
⚠️

Common BugForgetting & in scanf is one of the most common beginner mistakes. scanf("%d", x) is wrong; scanf("%d", &x) is correct. The exception: char arrays (strings) don't need &.

Day 04 Operators & Expressions Beginner

C has arithmetic, relational, logical, bitwise, and assignment operators. Integer division truncates (5/2 == 2). Bitwise operators are crucial for embedded/systems work and will appear throughout this roadmap.

// Bitwise operators — frequently used in C
int flags = 0b00001010;  // binary literal (C99+)
flags |= (1 << 2);   // set bit 2
flags &= ~(1 << 1);  // clear bit 1
flags ^= (1 << 3);   // toggle bit 3
int bit2 = (flags >> 2) & 1; // read bit 2

Day 11Multiple Return Values & Output ParametersBeginner

C functions return one value. To "return" multiple values, pass pointers as output parameters — the function writes results through them. This pattern appears everywhere in C APIs.

void minmax(int *arr, int n, int *min, int *max) {
    *min = *max = arr[0];
    for (int i = 1; i < n; i++) {
        if (arr[i] < *min) *min = arr[i];
        if (arr[i] > *max) *max = arr[i];
    }
}

Week 04 Pointers — The Heart of C Addresses, dereferencing, pointer arithmetic, function pointers +
Day 22Pointer Basics — Address & DereferenceIntermediate

A pointer stores a memory address. &x gives the address of x. *p dereferences — follows the pointer to read/write the value. This is C's most important concept.

🐍

From PythonPython variables are all references under the hood, but you never see addresses. In C, you work directly with addresses. This is intimidating at first but gives you total control over memory layout and performance.

int x = 42;
int *p = &x;           // p holds the address of x
printf("%p\n", (void*)p); // print the address
printf("%d\n", *p);     // dereference: prints 42
*p = 100;              // modify x via pointer
printf("%d\n", x);      // prints 100

Exercises

  3. Write void swap(int *a, int *b) and verify with prints before and after the call.
  4. Demonstrate that passing a variable (not a pointer) to a function does NOT modify the original.
  5. Debug this: int *p; *p = 5; — what's wrong and why is this dangerous?
🚨

Uninitialized Pointerint *p; *p = 5; is undefined behaviour — p contains garbage, you're writing to a random memory address. Always initialize pointers: int *p = NULL; or int *p = &someVar;

Week 05 Dynamic Memory Management malloc, calloc, realloc, free, memory leaks, valgrind +
Day 29Stack vs Heap — The Mental ModelIntermediate

Local variables live on the stack — fast, limited (1–8 MB), automatically freed when the function returns. malloc allocates on the heap — large, slow, persists until you free() it. Mixing these up causes crashes.

🐍

From PythonPython's garbage collector handles memory. In C, you are the garbage collector. Every malloc needs a matching free. Forget, and you have a memory leak. Free twice, and you have undefined behaviour.

Exercises

  2. Demonstrate that returning a pointer to a local variable is a bug: write the function, observe the garbage output.
  3. Allocate an array of 5 ints with malloc, fill it, print it, free it. Always check if (ptr == NULL) after malloc.
  4. Use calloc(10, sizeof(int)) — verify all elements are zero-initialised (unlike malloc).
  5. Write a "safe malloc" wrapper that prints an error and exits if allocation fails.
Day 30malloc, calloc, reallocIntermediate

realloc resizes an existing allocation. It may move the memory — never use the old pointer after calling realloc. Use the return value.

int *arr = (int*)malloc(5 * sizeof(int));
if (!arr) { perror("malloc"); return 1; }

// Resize to 10 ints
int *tmp = (int*)realloc(arr, 10 * sizeof(int));
if (!tmp) { free(arr); return 1; }  // don't leak on failure!
arr = tmp;

free(arr);
arr = NULL;  // nullify to prevent use-after-free

Learning Efficiency Tips

How to get the most out of every day.

  • ⌨️
    Type every line of code.Never copy-paste from this guide or anywhere else. The muscle memory of typing C syntax — semicolons, braces, pointer syntax — is real and it matters. Your fingers will start typing correct C before your brain knows why.
  • 🔴
    Make things break on purpose.After each exercise works, try to break it: remove a semicolon, pass NULL, write past the array, forget to free. Understanding failure modes is as valuable as writing correct code. Read every error message carefully.
  • 📐
    Draw memory diagrams.For every pointer exercise, draw a box-and-arrow diagram: boxes for variables/memory cells, arrows for pointers. This is not optional for Weeks 4–5. If you can't draw it, you don't understand it.
  • 🔄
    Compile often, change little.Work in small increments. Write 5–10 lines, compile, fix errors, test. Don't write 100 lines and then compile — you'll have 15 errors and no idea where to start.
  • 🐛
    Read compiler errors front to back.GCC/Clang errors include the file name, line number, and a description. Fix the FIRST error, recompile — subsequent errors often cascade from the first. Never try to fix all errors at once.
  • 🧪
    Test every function independently.Write small main() test stubs for each function before integrating. If isPrime(17) works in isolation but fails in your larger program, the bug is in the integration, not the function.
  • 🌙
    Sleep is part of the practice.Memory consolidation happens during sleep. A hard concept that makes no sense on Tuesday often clicks on Wednesday morning. Don't pull all-nighters — spread the 90 days evenly and actually sleep.
  • 🔍
    Use valgrind from Day 29 onward, always.Make it a habit: never submit or move on from a project until valgrind says zero leaks, zero errors. This habit alone separates intermediate C programmers from beginners.

Common Beginner Mistakes

The bugs that will definitely find you — know them before they do.

  • 🚨
    Forgetting & in scanf.scanf("%d", x) is wrong — it passes the value, not the address. Always: scanf("%d", &x). Exception: char arrays don't need & because they already decay to a pointer.
  • 🚨
    Off-by-one in arrays.An array of size n has valid indices 0 to n-1. Writing to index n is a buffer overflow. This is silent, dangerous, and will crash your program at a random later point — not at the line that caused it.
  • 🚨
    Using = instead of == in conditions.if (x = 5) assigns 5 to x (always true). if (x == 5) compares. Use -Wall — GCC warns about this. Some people write if (5 == x) (Yoda conditions) to prevent it.
  • 🚨
    Returning a pointer to a local variable.int* f() { int x = 5; return &x; } — x is destroyed when f() returns. The pointer dangling into freed stack space. Always allocate on the heap (malloc) if you need to return memory.
  • 🚨
    Integer division when you want float division.5 / 2 == 2 in C (truncates). Cast at least one operand: (double)5 / 2 == 2.5. A very common source of subtle bugs in calculations.
  • 🚨
    Missing break in switch.Without break, execution falls through to the next case. This is intentional sometimes, but usually a bug. Always add break unless fall-through is explicitly intended (and comment it).
  • 🚨
    Modifying a string literal.char *s = "hello"; s[0] = 'H'; is undefined behaviour — string literals are in read-only memory. Use char s[] = "hello"; if you need to modify the string.
  • 🚨
    Using freed memory.After free(p), any access to *p is undefined behaviour. Set p = NULL immediately after freeing. A NULL dereference crashes with a clear segfault; a use-after-free corrupts silently.
  • 🚨
    Confusing ++p and p++ with *p.*p++ dereferences p then increments the pointer (not the value). (*p)++ dereferences and increments the value. *++p increments p first then dereferences. Know all three.
  • 🚨
    Using gets() or strcpy() without bounds checking.gets() was removed in C11. strcpy(dst, src) doesn't check if dst is large enough. Always use fgets(buf, sizeof(buf), stdin) and strncpy(dst, src, sizeof(dst)-1).
